Five Years On

(Talk Structure)

Five years ago, mob programming was something we hadn’t heard about at ctm. Today, most teams have tried it and many have found that it helps them.

Some have been doing it for years, some are still new to it.

I’m going to tell the story of three teams that started mobbing What went well, what went not so well What was learned

I’m hoping that these will encourage you to give mobbing a go, or if you already have, find ways to make your mobs even better.

Starting To Mob

Why don’t we just work together?

What is Mob Programming?

Mob programming is pair programming taken to the next level.

🙂🖥🙂

Pair programming is when two people work together on a task at the same time. One person types, or “drives”. Usually the pair take turns to drive.

🙂🙂🙂🙂 🖥

Mob programming adds more people.

(Continuing 2)

with a new project:

before: hard to get agreement on the many decisions (which libraries, test framework, ...), so big upfront meetings

after: we make decisions as we go, everyone is here

before: hard to keep code consistent, re naming

after: we spot when someone is doing something different, have a conversation before it happens

before: hard to keep quality consistently high with different skills/experiences in team, code reviews are tricky

after: mobbing gets best out of everyone (everyone has their strengths)

after: easier conversation than code review

(Difficulties)

But we had some difficulties:

Mobbing is intense

Tiredness

We’d find ourselves going too long, people became tired Answer was to enforce regular breaks

(Difficulties 2)

Safety

It can be intimidating driving when people are watching what you do and giving instructions Kindness, patience very important Need to create a safe place for people to do their best

(Summary)

we’ve learned how to mob

and we love it

Mobbing amplifies all the good things about pairing But if there are issues, these can be amplified too

mobbing works by shortening feedback loops so we start to question some of our practices

retros – we tend to bring things up in real-time instead

code reviews – we’ve already review it
